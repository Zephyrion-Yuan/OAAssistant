OAAssistant Graph 设计评估

当前 LangGraph 设计在「工作流 ↔ 智能体」谱系上的客观定位 · 是否 AI-native · 可改进点  |  2026-06-05

⬆ 更新(2026-06-06):P1 / P2 / P3 已落地

本报告下方的「现状评估」描述的是评估当日(L1 工作流)的状态。此后已按报告建议实现了 agentic 外壳 + 确定性内核: P1 ReAct 对话前端(agent/:create_react_agent + 只读工具 query_pdm/inventory/resolve_wbs/query_wbs + emit_demand,/api/agent-chat,前端「AI 下单」)、 P2 复合请求规划(一句话采购+归还 → 按目标拆组、forced_goal 各自跑图、草稿聚合)、 P3 自适应采集(物料码查无→按名重查)。理解/规划层已从 L1 提升到 L3;执行/业务规则/安全仍是确定性内核,**永不提交**这条红线一行未改。 下文保留作为「为什么这么改」的依据。当前架构以 ARCHITECTURE.md §4.7/4.8 为准。

一句话结论(评估当日)

当前设计是一套「LLM 增强的工作流(workflow)」,不是「智能体(agent)」。客观依据只有一条、且决定性: acquire 图的 13 个节点之间,每一条边都是 state 的确定性函数 —— LLM 从不决定下一个节点走向。 这正是 LangChain/Anthropic 对 workflow 的定义(「predetermined code paths」)。所以你的观察是事实正确的:它在控制流层面与传统自动化(RPA)没有本质区别,LLM 只是被当作 4 个离散「算子」(分类 / 判断 / 抽取 / 写引导),而非「大脑」。

但「不是 agent」并不等于「设计得不好」。对这个领域(必须永不自动提交、永不臆造选择器、可审计、业务规则要精确)而言,工作流恰恰是被官方文档推荐的、更合理的选择。真正值得做的不是把执行核 agent 化,而是在"理解 / 规划"这一安全的前端加一层真正的 agent —— 详见下文分级建议。

1 · 评估依据(主源)

本报告刻意只引官方/一手定义,避免顺着主观感受走:

  • LangChain《Workflows and agents》(canonical 概念文档):workflow = 「预定代码路径、按既定顺序运行」;agent = 「动态地自定义自己的流程与工具使用」。命名模式:prompt chaining / parallelization / routing / orchestrator-worker / evaluator-optimizer。
  • Anthropic《Building Effective Agents》(上面分类法的源头):workflow 适合「可预测、定义良好」的任务;agent 适合「需要灵活性与模型驱动决策、且要规模化」的场景;并强调「先找最简方案,确有必要再加复杂度」,因为「agent 用延迟和成本换取任务表现」。
  • LangGraph agent 架构谱系:router(LLM 做一次分类、控制流仍固定)→ tool-calling/ReAct(LLM 自主循环决定调哪个工具)→ supervisor/multi-agent。官方建议:标准 agent 用 create_agent 预制 ReAct;只有需要自定义控制流 / HIL / 持久化时才降到底层 LangGraph 手写图。

关键点:在这套权威话术里,“AI-native / agentic” 的硬指标是「LLM 是否掌控轨迹(谁决定下一步)」,而不是「用没用 LLM」。下面用这把尺子量当前设计。

3 · 它是不是 AI-native?

把 “AI-native” 定义清楚再回答,否则只是吵词:一个系统的 AI-native 程度 = LLM 在多大程度上"掌控轨迹"(决定下一步做什么),而不是被当成固定代码里的一个函数来调。

诚实的双面回答:

  • 按"agent = LLM 掌控轨迹"的硬标准:当前设计不是 AI-native。 你感觉到的"和自动化软件区分度不高",在控制流层面是客观成立的 —— 图是一台确定性状态机,LLM 被关在 4 个节点内部当算子。
  • 但"AI-native"不是质量勋章。 按 Anthropic/LangChain 的同一套文档,workflow 对"可预测、规则明确"的任务本就是更优解,而本系统恰是这类:要可审计、要可复现、业务分配规则(412>89>458)必须精确、还有一条不可让步的红线——永不自动提交、永不让 LLM 生成选择器。这条红线从根本上排斥一个"自主决定动作"的高自治 agent。

所以:「不是 agent」是这个项目深思熟虑的、且与最佳实践一致的取舍,不是缺陷。 你的观察对,但"因此应该把它 agent 化"这个结论并不自动成立 —— 要分清"哪里该加 agency、哪里绝不能加"。

4 · 关键判断:缺陷,还是合理?(逐项)

你可能觉得"太死板"的地方客观判断
大量固定 conditional edges合理。 它们绝大多数是校验闸门(物料是否存在、单位是否一致、是否登录、字段是否齐),不是"语义模糊、需要模型判断的岔路"。校验闸门就该是确定性的——用 LLM 反而引入不可靠与不可审计。
分配走死算法而非 LLM合理,且应保持。 412>89>458 + 共享池扣减是业务规则,要的是精确、可复算、可审计。LLM 算分配 = 引入幻觉风险且无法过审。
填单步骤全写死(选择器/模态序列)必须如此(硬红线)。 仓库的根本约束就是 LLM 永不操作浏览器、永不生成选择器。这部分越"哑"越安全。
纠错/分诊也半确定性恰当。 assist 已经是"确定性先筛 + LLM 啃残差 + 写引导",这是目前最 AI-native 的部分;再激进会牺牲可控性。
整条链是一条写死的流水线这是唯一真正"过度死板"的地方。 见 §5 —— 前端理解/规划层确实可以、也应该更 agentic。

5 · 真正可以更 AI-native 的地方(安全前提下)

原则:把 agency 加在"理解与规划"(读多写少、可回退),把确定性留给"执行、业务规则、安全红线"。 这正是 LangGraph 推荐的组合方式——让一个 agent 去编排既有的确定性工作流(把每个 flow 当工具/子图)。

P1把对话前端换成真正的 ReAct agent(收益最大、且安全)

现状:右侧基本是"填表",左侧聊天主要是进度+纠错通道。用户其实不能用自由的多意图自然语言下单,模型也不能自己决定先查什么。

改法:用 create_agent 在入口放一个 ReAct agent,只给它只读工具:query_pdm / inventory_query / resolve_wbs / query_wbs,外加一个"产出结构化需求"的工具。它自主地:澄清提问 → 决定查哪些 → 组装出 demandRows → 交给现有确定性填单骨架(save=false)。

为什么安全: 这个 agent 能调的全是只读/可回退动作,真正的写操作仍走那条"哑"的确定性管线、且永不提交。AI-native 度从 L1 跳到 L3,但红线一条没碰。

P2用 supervisor/planner 取代写死的"classify_goal → 单一流水线"

现状:意图只在 acquire/return 二选一,且后续是一条固定管线;复合/新型请求(又采购又归还、只想先看库存再决定、信息不全)无法表达。

改法:把四个 flow + PDM/库存查询登记成"工具",让一个 planner 节点(LLM)对无法预定义的请求决定调用顺序与组合 —— 这正是文档里 orchestrator-worker「subtasks cannot be predefined」的适用场景。确定性分配算法仍作为它的一个工具被调用,不被替换。

边界: planner 只决定"调哪些只读/装配工具、什么顺序",不决定"怎么点页面、要不要提交"。

P3把单次信息采集升级为有界的自适应循环

现状:pdm_enrich / inventory_query 都是固定单次。改法:允许模型在结果歧义时自适应重查(码查不到就按名字查、库存按不同仓/项目维度再拉一次)。这是只读、可回退、天然带 LangGraph 循环的 agentic 行为,安全且更能"自己解决问题"。

不该 agent 化、应坚决保持确定性的部分

  • 分配/路由业务规则(412>89>458、按 WBS 分桶)—— 要精确、可审计。
  • 填单执行(选择器、模态、保存)—— 硬红线,永不交给 LLM。
  • 校验闸门(物料存在、单位一致、登录、字段齐)—— 确定性更可靠。
  • 永不提交/审批/付款/删除 —— 任何 agent 化都不得触碰这条边界。

6 · LangGraph 习惯用法层面的小观察

点现状 → 评价
人机协同(HIL)acquire 的纠错用"从 START 重入 + 全图重跑"而非 LangGraph 原生 interrupt()/Command(resume=…)。取舍可理解(前端无关、无状态),但代价是每次纠错重跑一遍 PDM/库存/LLM;若想省算力/更地道,可评估在阻塞点 interrupt 原地恢复。
并行(Send API)多草稿在 execute_plan 内串行循环而非并行 worker。正确:OA 单页有状态、必须串行;此处不该用 Send。
子图 / create_agent四个 flow 目前是"一个节点按 workflow_id 分发";若走 P1/P2,把它们包成子图/工具更利于被 agent 编排,也更符合 1.0 的 create_agent 习惯。
双图(single/acquire)single 模式仍带 understand/diagnose 的启发式兜底;acquire 是主力。两套并存是历史包袱,长期可收敛到一套"agentic 前端 + 确定性内核"。

落地次序(每步都可独立验证、且不破坏现有离线测试):

  1. 先把 query_pdm/inventory_query/resolve_wbs/query_wbs 暴露成 LangChain tool(只读),写一个 create_agent 装配前端,产出 demandRows → 喂现有 acquire 图。(P1,最高性价比)
  2. 给前端 agent 加澄清式提问能力(它自己决定何时追问),替换"必须填满右侧表"。
  3. 再上 planner/supervisor 处理复合请求(P2);确定性分配算法作为工具被调用。
  4. 评估把 HIL 从"重入重跑"迁到原生 interrupt(省算力、更地道)。

8 · 总评分

维度评分说明
工程质量(状态/检查点/审计/安全闸/测试)强可恢复、可审计、10 个离线测试、安全红线硬编码——作为 workflow 是高质量的。
AI-native / agentic 程度低(L1,局部 L2)控制流 100% 确定性,LLM 仅作算子。你的观察成立。
对本领域的适配度高安全/审计/确定性要求下,workflow 是被文档推荐的更优解;不该盲目 agent 化执行核。
提升 AI-native 的空间明确且安全前端理解/规划层(P1/P2/P3)可从 L1 提到 L3,红线一条不碰。

净结论:不是 AI-native,但当前的"不 agent 化"主要是对的;短板集中在前端理解/规划过于流水线化。真正的升级路径是"agentic 外壳 + 确定性内核",而不是把执行核交给模型。
