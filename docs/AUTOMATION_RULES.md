# 纯代码自动化规则

## 原则

- 程序内部不调用大模型，不做语义推理。
- 字段匹配只使用确定性规则：label、name、placeholder、id、selector、配置文件。
- OA/PDM 登录只依赖 Edge 浏览器 Cookie/Profile，授权过期时提示用户扫码。
- 保存草稿可以自动执行，提交审批不自动执行。

## OA 执行模式

1. 打开指定 workflow URL。
2. 检测是否进入登录页；如需要登录，截图并返回二维码/登录页面。
3. 扫描页面字段、按钮、表格、XHR/fetch 请求。
4. 根据扫描结果和 `config/field-mappings*.json` 定位字段。
5. 填写文本、下拉、组合框和附件。
6. 点击“保存草稿/暂存/保存”类按钮，明确排除“提交”类按钮。
7. 返回执行结果、失败字段、页面接口请求和最终字段状态。

## PDM 执行模式

1. 打开 PDM 物料页面。
2. 检测是否进入登录页；如需要登录，截图并返回二维码/登录页面。
3. 根据查询类型选择“物料编码/物料号”或“物料名称”输入框。
4. 输入查询值并点击“查询/搜索”按钮。
5. 读取页面表格表头与行数据，按表头组装返回。

## PDM 登录诊断

如果 PDM 页面没有二维码，先使用 `/api/pdm/login-diagnose` 判断登录机制。诊断只观察自动化 Edge 页面自身的跳转和请求：

- 是否跳到 `/auth/login`。
- 是否出现账号密码、企业微信/钉钉、SSO、CAS 等按钮。
- 是否调用了会话检查接口。
- 是否尝试打开外部协议或跳转到统一认证地址。

不允许把程序做成钉钉本地请求抓包器，也不允许导出钉钉或浏览器 Cookie 值。正确方案是让 IT 提供正式 SSO/OAuth/CAS 接入参数，或让用户在同一个 Edge Profile 中人工登录一次并由浏览器缓存授权。

可接受的 PDM 登录接入方式：

- 用户在工具启动的 Edge Profile 中人工完成一次 PDM 登录，后续复用浏览器授权。
- 用户从钉钉工作台打开 PDM 后，把实际入口 URL 填入前端的“PDM 入口 URL”或 API 的 `url` 参数。
- 用户在本机前端临时粘贴一次性 SSO/callback URL，让自动化 Edge 打开并落下 PDM 会话 Cookie；工具返回值和诊断日志必须脱敏授权参数。
- IT 提供正式的企业 SSO/OAuth/CAS 接入说明，由程序按官方接口获取授权。

不接受的方式：

- 抓取钉钉本地 HTTP/IPC 请求以提取 authCode、token 或 ticket。
- 读取钉钉进程内存、数据库、缓存文件。
- 从浏览器或系统 Cookie 存储中导出 Cookie 值再注入。

工具提供 `/api/pdm/auth-probe` 用于探测 PDM 页面公开暴露的登录路由和企业 SSO 按钮。这属于页面级自动化，不涉及本地凭证拦截。
