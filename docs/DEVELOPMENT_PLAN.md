# 开发计划

## 目标

构建一个本地规则自动化执行服务，通过 Edge 访问 MEGA OA/PDM，并提供前端页面和 HTTP API 两种入口，供人工操作、脚本或 MCP 客户端调用。

程序内部不使用 AI 推理。字段发现、字段匹配、PDM 查询、附件上传、保存草稿均通过代码规则和配置文件执行。

## 边界

- 不自动提交审批，只保存草稿，保留用户 double-check。
- 登录需要用户扫码授权，程序只缓存浏览器登录态。
- 首期只覆盖已有物料编码的标准品物料。
- OA 字段映射先通过扫描发现，再逐步固化为稳定脚本。

## 阶段

1. P0 骨架
   - Edge 持久化 Profile 与登录失效检测。
   - 前端控制台和 API 服务。
   - OA 页面扫描：字段、必填状态、按钮、XHR/fetch 请求。
   - PDM 查询：按编码/名称输入、读取表格。

2. P1 OA 固化
   - 对 workflow 458/412/414/89 分别跑扫描。
   - 整理必填、选填、联动字段。
   - 为项目号 `C2-0225002`、WBS `C2-0225002.06.01` 建立测试填单配置。
   - 固化附件上传与保存草稿按钮识别。

3. P2 业务规则
   - 物料编码 -> 物料组 -> 采购组 -> 交期映射。
   - 条件必填与格式校验。
   - Excel 附件模板生成。
   - 用户偏好与项目默认值缓存。

4. P3 外部编排
   - 封装更稳定的 API schema。
   - 增加任务队列、审计日志、失败截图。
   - 增加 MCP server 或 tool wrapper，让外部自动化客户端可直接调用。

## 需要人工确认的信息

- 四个 OA workflow 分别对应的业务名称。
- 保存草稿按钮真实文案。
- 附件上传控件与 Excel 模板要求。
- PDM 查询页的搜索字段名、表格分页规则。
- 是否允许长期使用独立 Edge Profile，还是必须复用系统默认 Profile。
