OA Assistant · Stage 1

把一条自然语言 + Excel 变成 OA「库存转储(workflow 89)」草稿；先用 PDM 校验物料，再由确定性脚本填单。
LLM 只负责「理解」，Playwright 负责「动手」，永不自动提交。

✅ Node 语法检查通过 ✅ Python 编译通过 ✅ 离线端到端 4/4 通过 ✅ CLI + 断点续跑可用

0一句话心智模型

🧠 Python(LangGraph) 是大脑 —— 理解你说什么、缺什么就追问、决定下一步。
🤝 Executor 契约是插座 —— 大脑只认这个插座，不关心背后是什么设备。
✋ Node(Playwright) 是手 —— 真正点按钮、填表格、读接口，已经写好、久经测试。

作为只懂 Python 的你：90% 的工作在 Python 那一层；Node 那层你基本当成一个「本地 HTTP API」用就行，本报告会逐文件标注。

2为什么是「Python + Node」两种语言？

现有项目里，4 套 OA 填单流程、PDM 查询、页面探索框架都已经用 Node + Playwright 写好并反复调通了（处理了大量弹窗、模态框、SAP 接口细节）。用 Python 重写一遍既浪费又有风险。

所以策略是：不动那只「手」，给它套一个 HTTP 接口；把新的「大脑」放在 Python —— 因为 LangGraph（状态机式 Agent 编排）是 Python 的主场。你只在 Python 里写逻辑，Node 当成一个本地服务调用即可。

分支（条件边）

check_slot：缺库存地点/WBS → ask（交互→回 understand 追问循环；无人值守→needs_input 可恢复终态）
pdm_enrich：物料编码不存在/未启用 → diagnose → needs_input
verify：ok → finalize；失败 → diagnose
diagnose（自愈大脑）：瞬时错误→retry 回 execute；登录失效→needs_login；结构漂移→转人工 escalate
ask 追问只发生在「输入阶段」；填单→保存草稿是无人值守。最终「提交审批」永远人工。

5Executor 契约 = 设备分离的关键

这是你那条「想把设备相关脚本和 Agent 工作流分开」需求的落地。Agent 图只 import 这个 Protocol：

# executors/base.py
class Executor(Protocol):
    def session_status(self) -> dict: ...
    def query_pdm(self, filters: dict) -> dict: ...
    def fill_stock_transfer(self, req: FillRequest) -> ExecutionResult: ...

三种实现，按 EXECUTOR 环境变量切换：

实现用途
HttpNodeExecutor调真实 Node 服务；Win/mac 差异藏这里
MockExecutor内存假 OA/PDM，离线测试/CI
(未来) Mac/PyPlaywright同契约，新实现即可，图不变

想加 mac 专用执行端、或换成纯 Python Playwright？只需新写一个满足契约的类，graph.py 不用动。

7Python 编排层 —— 你的主战场 Python

目录 orchestrator/oa_orchestrator/。每个文件职责单一，先看 graph.py 串起全局，再看具体节点。

核心骨架

graph.py把 11 个节点用条件边连成状态机；挂 SqliteSaver 检查点。想看全局先看这里。
run.pyCLI 入口。解析参数 → 建执行端/检查点/图 → 流式跑 → 打印结果 + 写 run.json。
state.pyGraphState（TypedDict）：在节点间传递的「公文包」+ 状态常量（done/needs_input…）。
schemas.pyPydantic 数据契约：BusinessInput / Intent / FillRequest / ExecutionResult / Diagnosis。
config.py读环境变量（每次调用都读，便于覆盖）：NODE_BASE_URL、DEEPSEEK_API_KEY、EXECUTOR…
store.py独立 SQLite business_inputs 表的读写（结构化 Excel 持久化）。
llm.pyDeepSeek 模型工厂 + 结构化输出助手；没 Key 时返回 None，让节点走规则兜底。

Executor 契约 executors/

executors/base.py契约本体（Protocol）+ get_executor 工厂。
executors/http_node.py用 httpx 调 Node 服务的真实实现。
executors/mock.py内存假 OA/PDM（含 2 个物料、2 个库存地点），离线测试用。

节点 nodes/（每个就是一个函数）

intake.py移植自 stock_transfer_excel.py openpyxl 解析 Excel → BusinessInput → 存表。含纯函数 parse_workbook() 便于单测。
understand.pyLLM 抽意图；无 Key 时用正则兜底（找「X仓」「D002」这种）。自然语言入口在这。
check_slot.py纯函数：算缺哪些必填槽（转出/转入库存地点、按移动类型需要的 WBS）。
ask.py缺槽时生成追问；交互模式用 LangGraph interrupt() 暂停等回答，无人值守则进 needs_input。
preflight.py经契约问登录态；未登录→needs_login（可续跑）。
pdm_enrich.py跨系统步：逐物料查 PDM 校验「存在且启用」，补全单位/名称；坏码→needs_input。
resolve_params.pyintent → 执行端参数（camelCase）。当前确定性映射，留了 LLM 消歧扩展位。
execute.py组 FillRequest，经契约调 fill_stock_transfer；摄取 actions/requestId。
verify.py核对是否走到「保存草稿」（看 requestId）。
diagnose.py自愈大脑：失败分类→瞬时重试/缺输入/登录失效/结构漂移转人工；受 max_retries 约束。
finalize.py写两级审计 run.json（图节点级 + Playwright 步级），定最终状态。
_common.py共享小工具（默认移动类型、是否需要 WBS、history 追加）。

其它

tests/smoke.py离线端到端测试（造样例 xlsx + MockExecutor 跑全图）。改完代码先跑它。

10怎么跑起来

⚠️ 仓库里的 .venv 是旧的 Windows venv（不能在 mac 用）。已为你新建 orchestrator/.venv（python3.12）。

A. 离线自测（最快，什么都不用准备）

orchestrator/.venv/bin/python orchestrator/tests/smoke.py
# 期望最后一行：ALL OFFLINE SMOKE TESTS PASSED

B. 用 Mock 后端跑 CLI（无真实 OA）

PYTHONPATH=orchestrator orchestrator/.venv/bin/python -m oa_orchestrator.run \
  --executor mock --excel /tmp/oa_sample.xlsx \
  --request "从设备零件仓 D002 转到成品仓 A001" --save

C. 接真实 Node + DeepSeek（go-live）

# 1) 配置 LLM
cp orchestrator/.env.example orchestrator/.env   # 填 DEEPSEEK_API_KEY

# 2) 启动 Node 执行端（登录态：在弹出的托管 Edge 里手动扫码登录一次）
MEGANT_EDGE_PROFILE_MODE=sso-handoff npm start

# 3) 驱动它（默认 dry-run 只填不存；加 --save 才保存草稿）
PYTHONPATH=orchestrator orchestrator/.venv/bin/python -m oa_orchestrator.run \
  --excel /path/req.xlsx --request "..." --dry-run

# 中途崩了？同 thread 续跑：
... -m oa_orchestrator.run --resume --thread st-xxxxxxxx

CLI 参数：--request --excel --thread --resume --save --dry-run --interactive --executor

12安全边界（写进了代码，不只是口头）

🔒 大脑碰不到敏感物

Python 图只和本机 Node API 通信，从不接触 cookie/token；Node 端 redaction.js 已脱敏 URL/响应。

喂给 LLM 的只有业务字段（物料编码/名称/数量/库存地点/WBS/中文请求），绝不喂 SSO 链接、token、截图。

✋ 永不自动提交

执行端最多保存草稿；填单函数本就 never click「提交」。图里没有任何提交/审批路径。

首跑默认 --dry-run（只填不存），验完再 --save。最终「提交审批」永远人工在浏览器完成。
