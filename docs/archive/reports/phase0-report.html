① 框架全景:这套系统在做什么

目标:用户一句自然语言 + 一张「需求」表 → 系统自动判断每个物料该走采购/出库/转储/入库哪条 OA 流程,填出草稿(永不提交)。判断依据是库存在哪个仓、属于哪个项目。

双层架构(硬约束)

LLM 只做理解(意图/单位换算/追问/失败归因),Playwright 是确定性的手,永不自动提交。两层之间只走 HTTP 契约。

层职责
Node「手」填单、查 PDM、查库存、新 WBS 主数据;托管 Edge;确定性
Python「脑」(LangGraph)意图理解、路由决策、补全、自检;只经 Executor 契约碰 Node

四流路由规则(已锁定)

库存情形流程
无库存 / 数量缺口458 采购
公共库仓 · 本项目专属仓412 出库
纯别项目专属仓(SOBKZ=Q)89 转储
用完归还(意图触发)414 入库
草稿最小单位 = (流, WBS)。 每物料按需求量分配(优先级 412>89>458),跨物料再按 WBS 分桶 —— 同一个流里不同 WBS 拆成不同草稿。WBS 高度绑定工厂/成本中心/采购人等信息,所以需要一个用户可维护的 WBS 主数据库(本次 Phase 0 的核心)。

4.1 src/automation/wbsRegistry.js — 数据底座(Node 持有的唯一真相源)

为什么 JSON 文件而非 SQLite:数据量小、零新依赖、跨平台。合并式 upsert 是关键设计 —— 部分更新不会抹掉未提交字段。

// 存储:.runtime/wbs-registry.json = { version, records: { "": {...} } }
// 原子写:先写临时文件再 rename,避免半截写入损坏。
function saveRegistry(registry){
  ensureDir(runtimeDir);
  const tmp = `${REGISTRY_PATH}.${process.pid}.tmp`;
  fs.writeFileSync(tmp, JSON.stringify(registry, null, 2), 'utf8');
  fs.renameSync(tmp, REGISTRY_PATH); // 原子替换
}

// 合并式 upsert:只有 input 里出现的字段才覆盖;已有记录的其余字段保留。
export function upsertWbs(input = {}){
  const wbsCode = normalizeText(input.wbsCode);
  if(!wbsCode) return { ok:false, error:'wbsCode is required.' };
  const registry = loadRegistry();
  const existing = registry.records[wbsCode];
  const record = existing ? { ...existing } : emptyRecordShape();
  for(const field of TEXT_FIELDS){
    if(input[field] !== undefined) record[field] = normalizeText(input[field]); // 只覆盖提供的字段
  }
  // ...offset/status 同理;createdAt 保留,updatedAt 刷新...
  registry.records[wbsCode] = record;
  saveRegistry(registry);
  return { ok:true, created:!existing, record };
}
实测捕获的 bug: 第一版 upsert 是整体替换,部分更新(只传 purchaser)会把 demandDateOffsetDays 抹成 null。改成合并式后,「改采购人」不再误伤其它字段。CRUD/归档/删除/字段保留均已逐项验证。

4.3 public/ — 控制面板 WBS 管理页

一个全宽 <section>:上方表单(增/改),下方列表(编辑/归档/删除)。两处刻意的安全/稳健选择:

  • DOM 安全渲染:表格用 createElement+textContent,不把用户输入塞进 innerHTML(防注入)。
  • 删除两步内联确认:第一次点「删除」变「确认删除?」(3 秒后复原),第二次才真删 —— 不用 confirm() 原生弹窗(原生弹窗会阻塞,且与浏览器自动化冲突)。
// app.js: 列表用 GET(无 payload→GET),变更用 POST(有 payload→POST)

# http_node.py —— 真实后端:POST /api/wbs/get
def query_wbs(self, wbs_code):
    body = httpx.post(self._url("/api/wbs/get"), json={"wbsCode": wbs_code}, ...).json()
    return body.get("record") if body.get("ok") else None

⑤ 数据流:WBS 主数据如何被读写

控制面板页(你手动维护) — POST /api/wbs/upsert·archive·delete → wbsRegistry.js ⇄ 原子读写 ⇄ .runtime/wbs-registry.json
prepare 节点(Python,Phase 1) — executor.query_wbs(wbs) → HttpNode — POST /api/wbs/get → wbsRegistry.js → {factory, costCenter, purchaser, stockLocation, offset...}

写只发生在控制面板(人);读发生在编排层填单前(机)。Python 永远不直接碰那个 JSON 文件 —— 它只认 query_wbs 契约,这样换设备/换存储都不影响图。

为什么先做 Phase 0

route_workflow 选完流后,prepare 要靠 WBS 主数据把每张草稿的工厂/成本中心/采购人/库存地点补齐,并据此生成 458 附件。没有这个数据底座,prepare 无据可查、458 附件生不出来。所以先把 registry + query_wbs 建好、验透,Phase 1 才能顺。

安全边界(贯穿):全程只存草稿、零提交/审批;WBS registry 是本地业务主数据,不含任何 SSO/cookie/token/密码/MFA;喂 LLM 的仍只有业务字段。本报告本地查看,可直接用浏览器打开。
